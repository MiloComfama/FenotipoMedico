01

Resumen ejecutivo

El prototipo tiene dos vistas Streamlit sobre una misma base de datos: el paciente completa un cuestionario conversacional (guiado por "Fénix") que alimenta su clasificación de fenotipo y recomendaciones; el equipo médico registra antropometría y laboratorios, y puede reclasificar por criterio clínico. Cada consulta queda numerada y trazable, con la regla de negocio de una cita de seguimiento cada 30 días.

La clasificación hoy corre sobre un motor de reglas transparente, diseñado para ser reemplazado — sin tocar la UI ni la base de datos — por el modelo entrenado que sale del pipeline de clustering en databricks/.

BaseIntake

Cuestionario conversacional

Dos estrategias intercambiables detrás de la misma interfaz:

  • LLMIntake — Claude Opus 4.8, valida formato y numera opciones.
  • ScriptedIntake — flujo guiado sin IA, respaldo si no hay clave.
Classifier

Clasificación de fenotipo

El dominio no sabe cuál implementación corre — solo consume predict():

  • RuleBasedClassifier — reglas clínicas transparentes (hoy).
  • TrainedClassifier — modelo de model.pkl (futuro).

05

Modelo de datos

SQLite + SQLAlchemy en el prototipo, con esquema normalizado y listo para migrar a SQL Server / Microsoft Fabric. Las respuestas del cuestionario y los laboratorios usan un modelo llave-valor (EAV), así el cuestionario evoluciona sin cambiar el esquema.

La clasificación se guarda dos veces por consulta — phenotype_model (sugerencia del clasificador) y phenotype_final (definitivo, editable por el médico) — para mantener trazabilidad entre modelo y criterio clínico.

06

Asistente conversacional — Fénix

Fénix conduce el cuestionario de ingreso en lenguaje natural, con reglas pensadas para que responder sea rápido y sin ambigüedad.

①  Presenta las opciones de preguntas single/multi siempre numeradas, para responder solo con el número.
②  Valida el formato de cada respuesta (opción válida, rango numérico, fecha AAAA-MM-DD...) antes de guardarla; si no es válida, explica qué formato se espera y repregunta.
③  Solo resuelve dudas sobre la pregunta pendiente actual — si la persona pregunta algo ajeno, la redirige amablemente y retoma la pregunta, para mantener el foco.
④  Corre sobre Claude Opus 4.8 vía Azure AI Foundry (o la API directa de Anthropic); si no hay credenciales configuradas, cae automáticamente al flujo guiado sin IA.

08

Hoja de ruta del clasificador

El clasificador vive detrás de una interfaz estable para poder evolucionar sin tocar la aplicación:

1

Reglas clínicas transparentes Hoy

RuleBasedClassifier pondera síntomas, antropometría, presión y laboratorios en dos ejes (cardiometabólico / digestivo).

2

Exploración de fenotipos emergentes En curso

Notebooks 01→04 en databricks/: ingesta, preprocesamiento, clustering KMeans y perfilamiento sobre el histórico de Comfama.

3

Modelo entrenado conectado Próximo

TrainedClassifier se activa automáticamente al detectar data/model.pkl, sin cambios en la UI ni la base de datos.

4

Migración de persistencia Futuro

El esquema normalizado (EAV para respuestas y laboratorios) está pensado para migrar de SQLite a SQL Server / Microsoft Fabric sin rediseño.

09

Cómo ejecutar

py -m venv .venv && .venv\Scripts\activate

Entorno virtual (opcional).

py -m pip install -r requirements.txt

Instala dependencias.

copy .env.example .env

Completa ANTHROPIC_API_KEY (+ ANTHROPIC_FOUNDRY_RESOURCE si usas Azure AI Foundry). Sin esto, corre el flujo guiado sin IA.

py scripts/seed_demo.py

Pacientes de demostración (opcional).

streamlit run app/main.py

Abre http://localhost:8501.

Pacientes demo: CC 1001 (Digestivo), CC 1002 (Cardiometabólico) · PIN consola médica: comfama.

⚠️ Prototipo con fines de demostración. Las recomendaciones no reemplazan la valoración médica. El manejo de datos clínicos está sujeto a la autorización del paciente y al marco de habeas data y políticas internas de Comfama.
